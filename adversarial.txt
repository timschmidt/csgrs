Adversarial test plan for csgrs
================================

Purpose
-------

This file is a deliberately aggressive test inventory for the csgrs geometry
suite. It is meant to guide unit tests, property tests, fuzz targets,
metamorphic tests, snapshot tests, sanitizer runs, and long-running stress
jobs. The goal is not only to verify expected examples, but to break every
assumption about coordinates, topology, tolerances, feature flags, imports,
exports, and algorithmic complexity.

Core pass/fail rules
--------------------

Every test should declare which behavior is required:

1. The operation returns a valid object and preserves the documented geometric
   invariant.
2. The operation returns an empty object when the input is invalid or
   geometrically void.
3. The operation returns a typed error where the API supports errors.
4. The operation panics only if the API explicitly documents panic behavior.
5. The operation must not hang, overflow, silently produce NaN from finite
   input, corrupt metadata, create invalid indices, or allocate unbounded
   memory.

Global invariants to check everywhere
-------------------------------------

- All finite-input public APIs should produce only finite coordinates,
  normals, areas, volumes, bounding boxes, indices, and quality metrics.
- No triangle index should refer beyond the vertex array.
- No exported face should have fewer than three distinct vertices unless the
  format explicitly permits a degenerate placeholder.
- Polygon normals should be finite, normalized where the API promises unit
  normals, and consistently flipped by inverse/mirror operations.
- `bounding_box()` should enclose every emitted vertex after transforms,
  booleans, imports, triangulation, smoothing, refinement, subdivision, and
  conversion.
- `invalidate_bounding_box()` should force recomputation and not change
  geometry.
- `center()` should place the bounding-box center near the origin for non-empty
  finite geometry.
- `float()` should place minimum z near zero for non-empty finite geometry.
- `inverse().inverse()` should be geometrically equivalent to the original.
- Identity transform should be a no-op up to tolerance.
- Transform composition should match sequential transforms up to tolerance.
- Boolean identity laws should hold: A union empty = A, A difference empty = A,
  A intersection empty = empty, A xor empty = A.
- Boolean annihilator laws should hold: A difference A = empty, A intersection
  A = A, A xor A = empty, A union A = A.
- Boolean commutativity should hold for union, intersection, and xor, but not
  difference.
- Boolean associativity should be checked for union and intersection on clean
  inputs, with tolerance-aware equivalence.
- All repeated operations should be deterministic when run twice in the same
  process with the same feature flags.
- Parallel and non-parallel builds should produce equivalent geometry modulo
  ordering.
- The hyperreal build should pass one explicit tolerance budget across feature
  sets; f32/f64 feature matrices are retired.
- Metadata should survive operations that document preservation, and should not
  be spuriously copied where semantics require new or filtered metadata.
- Empty meshes, empty sketches, empty NURBS, and exact hypermesh adapters
  should not crash downstream APIs.
- Every invariant above should be encoded in reusable proptest assertion
  helpers so generated Mesh, Sketch, Polygon, Plane, Transform, I/O, wasm, and
  toolpath cases all check the same safety rules.
- Every panic, hang, non-finite output, invalid index, and minimized proptest
  failure should be promoted into the deterministic regression corpus and, if
  byte-oriented, seeded into the fuzz corpus.

Proptest and fuzzing strategy
-----------------------------

Use deterministic example tests for the hand-written edge cases in this file,
proptest for structured and shrinking-aware geometry generation, and
coverage-guided fuzzing for byte streams, API operation sequences, and invalid
or semi-structured inputs.

Proptest should be used throughout for:

- scalar, vector, angle, and matrix generation with explicit finite and
  non-finite strategies
- valid convex polygons, concave simple polygons, invalid self-crossing
  polygons, and polygons with holes
- valid watertight meshes from primitives and intentionally invalid meshes
  assembled from arbitrary polygons
- boolean expression trees over Mesh and Sketch, with closed solids audited
  through hypermesh
- transform sequences and inverse/round-trip checks
- import/export round trips whose input model can be represented as structured
  Rust values
- metadata preservation and filtering checks
- feature-flag differential checks, including parallel vs sequential output
  normalization under the shared hyperreal scalar model
- operation-chain generation, where a generated object is passed through
  multiple public APIs before invariant checks

Fuzzing should be used throughout for:

- parsers and importers: OBJ, SVG, DXF, Gerber, images, font bytes, and any
  future text/binary formats
- exporters with hostile names, metadata, comments, units, and colors
- wasm boundary inputs where JavaScript can supply malformed arrays and values
- operation bytecode interpreters that turn bytes into constructor choices,
  transforms, booleans, offsets, slicing planes, and export calls
- polygon and polyhedron constructors fed by raw coordinate/index streams
- matrix and plane splitting paths that have historically fragile tolerance
  behavior
- crash-only harnesses plus semantic harnesses that reject NaN output, invalid
  indices, bad bounding boxes, and non-terminating reductions

Required proptest/fuzz hygiene:

- Store `ProptestConfig` seeds for every CI failure.
- Use low case counts in fast CI, higher case counts in medium CI, and
  long-running fuzz jobs outside the fast path.
- Prefer small generated cases that shrink well, then add separate stress
  strategies with explicit memory and time budgets.
- Convert every minimized proptest case and every minimized fuzz artifact into
  a named regression fixture.
- Run fuzz targets with sanitizer builds where practical.
- Keep native Rust, wasm, parallel, and hyper geometry adapters fed from the
  same canonical corpora where possible.

Tolerance boundary cases
------------------------

- Points exactly equal.
- Points separated by tolerance() / 1024, tolerance() / 16, tolerance() / 2,
  tolerance(), tolerance() * 2, tolerance() * 16, tolerance() * 1024.
- Planes with offsets differing by the same tolerance ladder.
- Coplanar polygons whose normals differ by the same tolerance ladder.
- Edges that almost touch but do not cross by tolerance() / 2.
- Edges that should cross only after coordinate normalization.
- Vertices that quantize into the same `VertexIndexMap` bucket.
- Vertices just outside the same `VertexIndexMap` bucket.
- Bounding boxes with min/max differences near zero.
- Classification of points exactly on a plane, barely in front, barely behind,
  and at enormous coordinates where tolerance is small relative to magnitude.

Run `union`, `difference`, `intersection`, and `xor` on Mesh and Sketch, then

- empty vs empty
- empty vs non-empty
- identical objects
- identical objects with reversed winding
- identical objects with different metadata
- disjoint objects far apart
- disjoint objects separated by tolerance() / 2
- tangent objects touching at one point
- tangent objects touching along one edge
- tangent objects touching along one face
- overlapping by tolerance() / 2
- overlapping by exactly tolerance()
- overlapping by tolerance() * 2
- one object fully inside another
- one object fully inside another but touching boundary
- thin shell against solid
- coplanar overlapping polygons
- coplanar non-overlapping polygons
- coplanar polygons with opposite normals
- nearly coplanar polygons
- rotated objects at irrational angles
- scaled objects with negative determinant transform
- transformed objects with extreme aspect ratios
- many small objects unioned sequentially
- many small objects unioned in balanced tree order

Metamorphic boolean checks:

- A union B equals B union A.
- A intersection B equals B intersection A.
- A xor B equals B xor A.
- A difference B generally differs from B difference A.
- A union (B union C) equals (A union B) union C for clean non-degenerate
  solids.
- A intersection (B intersection C) equals (A intersection B) intersection C.
- A difference (B union C) equals (A difference B) difference C for clean
  solids.
- inverse(A union B) should align with inverse(A) intersection inverse(B) if
  inverse is intended as complement over the same universe; document any
  semantic limitations.
- Boolean outputs should be stable after `triangulate` and `renormalize`.
- Add proptest boolean expression trees with bounded depth, generated
  primitive leaves, generated transforms between leaves, and normalized
  equality checks for algebraic laws.
- Add fuzz operation-sequence targets that decode a stack-machine language:
  push primitive, transform, union, difference, intersection, xor, inverse,
  triangulate, export, and invariant-check.

I/O export attacks
------------------

Target STL ASCII/binary, OBJ, PLY, AMF, glTF, DXF, SVG, and Gerber exporters.

Common export cases:

- Empty geometry.
- One triangle.
- Degenerate triangles.
- Non-manifold mesh.
- Mesh with NaN coordinates.
- Mesh with infinity coordinates.
- Mesh with negative zero coordinates.
- Huge coordinate values.
- Very small coordinate values.
- Object names empty, very long, whitespace only, unicode, XML/JSON/control
  characters, path-looking strings, newline injection strings.
- Comments empty, very long, unicode, control characters, newline injection.
- Units empty, standard values, unknown values, XML injection strings.
- Metadata with special characters.
- Export then parse with a strict external parser where possible.
- Export should not produce invalid UTF-8 for text formats.
- Export should not write indices outside declared vertex counts.
- Export should not silently truncate large finite values without documenting
  precision loss.
- Add proptest strategies for hostile but valid object names, metadata,
  comments, units, colors, and small generated meshes.

  triangle soups, then validate text/binary exporters for crashes and basic
  structural correctness.

Format-specific export cases:

- STL ASCII: names containing `solid`, `endsolid`, newlines, and binary-looking
  bytes; normals for degenerate triangles.
- STL binary: triangle count overflow guard; header length exactly 80 bytes;
  no accidental ASCII-only assumptions.
- OBJ: object names with spaces and slashes; negative zero formatting; repeated
  vertices; face winding; no out-of-range face references.
- PLY: header count matches body; comments with newlines sanitized; property
  order stable.
- AMF: XML escaping for object name, units, metadata, and color fields; invalid
  color values negative, > 1, NaN.
- glTF: valid JSON; base64 buffer length matches accessor counts; min/max
  accessors finite; names escaped; empty geometry valid or rejected clearly.
- DXF: empty layers; huge coordinates; arcs/lines/polylines if exported.
- SVG: path data finite; viewBox around empty or huge geometry; transform
  preservation if supported.
- Gerber: units inch/mm; coordinate precision extremes; polarity transitions;
  step-repeat; aperture holes; unsupported geometry ignored or errored
  predictably.

Property-based test generators
------------------------------

Use the `proptest` crate for this entire section. These generators should live
in shared test support modules so unit tests, integration tests, differential
tests, and regression-minimization workflows all use the same strategies.

Generate structured valid data:

- Convex polygons from sorted random angles.
- Star-shaped simple polygons from radial functions.
- Axis-aligned rectangles and boxes.
- Random transforms with bounded finite scale.
- Clean watertight triangle meshes from primitives.
- Clean sketches with non-overlapping holes.
- Random CSG expression trees over clean primitives.

Generate malformed data:

- Shuffled polygon vertices.
- Random duplicate point injection.
- Random spike insertion.
- Random coordinate replacement with NaN/infinity.
- Random edge collapse.
- Random winding reversal.
- Random face index corruption for import-like constructors.
- Random transform matrix corruption.
- Random metadata mutation.

Metamorphic properties:

- Transform identity is no-op.
- Transform composition equals sequential application.
- Translate by v then -v is no-op.
- Rotate by a then -a is no-op within tolerance.
- Scale by s then 1/s is no-op for nonzero finite s.
- Mirror twice is no-op.
- Triangulate area/volume equals source area/volume within tolerance for valid
  inputs.
- Export-import-export is stable after normalization for supported formats.
- Parallel and sequential outputs have same normalized triangle set.
- Hyperreal outputs remain within the same finite corpus envelope across
  feature sets.

Concrete "try to break everything" operation chains
---------------------------------------------------

- Invalid polygon -> Sketch::polygon -> triangulate -> extrude -> boolean with
  cube -> export all formats.
- Self-crossing Sketch -> offset inward -> offset outward -> extrude -> slice
  through every axis plane.
- Cube -> scale(0, 1, 1) -> union sphere -> triangulate -> mass_properties.
- Sphere segments 3 -> subdivide -> smooth -> remove poor triangles -> export.
- Mesh with duplicate opposite polygons -> BSP build -> inverse -> union cube.
- Huge cuboid -> center -> rotate tiny angle -> boolean with tiny sphere.
- Tiny cuboid -> translate huge -> center -> float -> export.
- Ring with touching hole -> triangulate -> extrude -> flatten -> to_svg.
- OBJ malformed import -> to_obj -> from_obj again.
- SVG arc-heavy path -> from_svg -> offset_rounded -> to_svg -> from_svg.
- Gerber with step repeat -> from_gerber -> to_gerber -> from_gerber.
- Text from corrupt font bytes -> extrude -> export if construction succeeds.
- Metaballs no-surface -> boolean with cube -> slice -> export.
- SDF closure returning NaN at one cell -> mesh quality -> export.
- Invalid Mesh -> hypermesh import rejection -> boolean -> transform -> exact
  validation if possible.
- wasm MeshJs from wrong-length arrays -> every method should throw or return
  documented empty output, never trap unexpectedly.

Definition of a normalized geometry comparison
----------------------------------------------

For equality-like assertions, avoid relying on raw polygon order. Normalize by:

- Rejecting non-finite coordinates first.
- Triangulating polygons if necessary.
- Sorting triangle vertices lexicographically after tolerance quantization.
- Sorting triangles lexicographically.
- Comparing quantized coordinates, normals where relevant, triangle count,
  bounding box, surface area, volume, and manifoldness.
- Allowing backend-specific tolerances for triangulation and exact-boundary
  adapter differences.

Final reminder
--------------

Do not only test "nice CAD input." The suite should include hostile inputs:
wrong counts, impossible parameters, self-intersections, non-manifold topology,
NaNs, infinities, enormous and subnormal scales, malformed files, broken JS
values, feature combinations, repeated operation chains, and cases constructed
to sit exactly on tolerance boundaries.
